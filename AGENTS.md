# 项目级开发约定

本文件是 `CyberKG-RAG` 项目的项目级协作约定。后续在本项目中开发、修改、评审代码时，优先遵守本文件。

## 1. 代码范围

1. 正式项目代码统一放在 `code/` 下。
2. 后端代码放在 `code/backend/`。
3. 前端代码放在 `code/frontend/`。
4. 数据、脚本、实验代码分别放在 `data/`、`scripts/`、`experiments/`。
5. 不在仓库根目录散放后端或前端源码。

## 2. 需求处理流程

当用户提出一个开发需求时，按以下流程执行：

1. 先分析需求，明确目标、影响范围、已有代码位置和风险点。
2. 给出改动方案，包括要改哪些模块、接口、数据结构、页面或测试。
3. 等用户确认方案后，再开始修改代码。
4. 修改完成后，检查一遍改动内容，确认没有明显遗漏、无关改动或路径错误。
5. 根据改动安排测试用例，说明应执行哪些测试、验证哪些场景。
6. 如果测试无法运行，要明确说明原因和剩余风险。

除非用户明确说“直接改”或“按你判断做”，否则不要跳过方案确认。

## 3. 代码风格

1. 代码要保持清晰、可维护，优先使用项目已有风格。
2. 关键逻辑需要添加中文注释，说明“为什么这样做”或“这段逻辑解决什么问题”。
3. 不给显而易见的语句添加废话注释。
4. 后端接口、数据模型、核心算法、复杂查询和 RAG 流程需要重点注释。
5. 前端组件中的复杂状态、图谱渲染逻辑、异常处理需要重点注释。
6. 避免无关重构，除非该重构是完成需求所必需的。

## 4. 前后端约定

1. 后端使用 FastAPI。
2. 前端使用 Vue3。
3. 前后端通过 API schema 对齐，不让前端直接依赖数据库字段细节。
4. 接口返回结构优先保持统一：

```json
{
  "success": true,
  "data": {},
  "message": ""
}
```

5. 前端页面需要考虑 loading、empty、error 三种状态。
6. 涉及图谱数据时，节点和边的结构要稳定，便于 ECharts 渲染。

## 5. 测试与检查

每次代码修改后，根据改动类型安排测试：

1. 后端接口：至少验证接口启动、成功返回、异常输入。
2. 数据处理：至少验证样例数据可重复生成、字段完整、无空实体。
3. 图谱查询：至少验证实体查询、1 跳邻居、多跳路径。
4. KG-RAG：至少验证有证据回答、无证据拒答、证据路径展示。
5. 前端页面：至少验证页面加载、接口失败、空数据和正常数据展示。
6. 模型实验：至少验证数据切分、训练入口、指标输出。

如果当前环境缺少依赖、数据库或 API Key，不能硬说测试通过，需要说明哪些检查已做、哪些测试未运行。

## 6. Git 操作约定

1. 不直接替用户执行 `git commit`、`git push`、`git reset` 等 Git 写操作，除非用户明确要求。
2. 如需提交或推送，只给出建议命令，由用户自己执行。
3. 不执行破坏性 Git 操作，例如 `git reset --hard`、强制推送、删除分支，除非用户明确要求并确认。
4. 若需要查看状态，可以运行只读命令，例如 `git status`、`git diff`、`git log`。

推荐提交命令格式：

```bash
git status
git add <files>
git commit -m "说明本次改动"
git push
```

## 7. 文档约定

1. README 只写运行环境、依赖、数据来源、运行步骤和项目状态。
2. 技术方案、开发计划、报告材料放在仓库根目录的 `文档/` 下。
3. 如果新增工程内部说明，可以放在 `code/docs/`。
4. 文档内容要尽量明确，避免只写口号式描述。

## 8. 安全与伦理

1. 本项目仅使用公开安全知识库数据。
2. 不对真实系统或未授权目标进行扫描、攻击、利用或验证。
3. 涉及漏洞、攻击技术、攻击路径时，表达应以防御、分析和教学为目的。
4. 报告和演示中需要保留伦理声明。
